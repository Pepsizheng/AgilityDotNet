* 协变  
参考该实现cs文件说明：[segregation](../AgilityDotNet/interfacesegregation/Program.cs);

* 逆变  
参考该实现cs文件说明：[segregation](../AgilityDotNet/interfacesegregation/Program.cs);

里氏替换不只是父类转子类，子类转父类的思想，还有原则指导，总结为严于律己，宽以待人。
解释为子类实现的时候，对于前置条件不能更严格，对于方法要返回前的后置条件不能更宽松。
同时对于只需要在类中或子类中修改的字段属性，不能被子类实现到外部类也可以修改。这条实现也叫做数据不变式。即某个条件规定后不变而不是指数据本身不变。

## 目前了解的敏捷流程

* 人员组成  
首先会根据项目要求，组建成一个团队负责该项目。项目成员包括如下分类：
1. 产品负责人：负责与客户需求对接，了解提出的需求和负责澄清团队对于需求的疑惑。
2. scrum主管：负责团队日常流程的把握，主要掌握整个流程的方向，解决团队遇到的项目外困难。
3. 开发人员：负责具体任务开发。
4. 测试人员：负责故事点开发交付物的测试。

* 整体流程
1. 产品负责人核对需求后，拉上团队一起对出该需求中的故事点。并给每个故事点赋值所需开发点数。
2. 产品负责人拉上团队，针对第一个冲刺需要完成的故事点与团队宣讲。产品负责人记录可能出现的需求增强点或答疑点。
3. 开发团队开始编程。
4. 开发人员进行同行评审。
5. 确认后开发人员提交代码至代码库。
6. 测试人员开始进行测试。
7. 演示会议。针对故事点先实现一个简单版本后给客户演示，及时获取反馈。
8. 回顾会议。什么做的好，什么做的不好，什么需要保持，什么需要改进，什么是意料之外的事情。
9. 记录演示会议中客户产生的新的故事点功能，再进行故事点评估后加入实现队列，继续2的步骤。

* 评价
1. 敏捷开发并没有减少开发测试的工作量，谨记。它只是一套应对需求变更的流程，对于实际开发的时候，项目安排进度紧张的话是否还需要划分很多会议，这点需要保留意见。
2. 与瀑布开发模式相比，敏捷是快速小而全的瀑布开发模式，很多文章说了很多华而不实的对比，但自己感觉并没有太大区别。敏捷只是把开发的编程变得更加频繁，因为会面对来自测试发现的bug，还有来自需求的变化。相比瀑布开发，开发一开始会面临需求的变化压力，等待实现之后，才会面临测试反馈bug的压力，从开发者的角度来看，当然更倾向于瀑布开发。
3. 敏捷开发提到的开发过程中的灵活性，其实瀑布开发也是这样，可以面对接口编程，你的软件的扩展性跟你采取的开发流程并没有太大关系。所以更多的提出所谓的敏捷开发目的，只是希望软件人员做的更快，响应更快，所以当有公司说采取的敏捷开发的流程，就要小心，可能是PUA的开始。
